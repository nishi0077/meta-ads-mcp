# Meta Ads MCP サーバー — 導入手順（テキスト版）

個人名・社名・アカウント固有の値は記載していません。手元の環境に合わせて置き換えてください。

---

## 前提条件

- **OS**: Windows / macOS / Linux のいずれか
- **Python**: 3.11 以上（`python --version` で確認）
- **Meta**: 広告を扱う権限のある Facebook / Meta アカウント
- **Meta Business**: 広告アカウントが紐づいていること（読み取りのみなら認証要件は緩い場合あり）
- **AI クライアント**: Claude Desktop、Claude Code、Cursor など MCP に対応したクライアントのいずれか

---

## 手順の全体像

1. リポジトリを手元に用意する  
2. Python 仮想環境を作り、依存パッケージを入れる  
3. Meta for Developers でアプリを用意し、OAuth 用のリダイレクト URI を登録する  
4. `.env` にアプリ ID とシークレットを書く  
5. 初回だけブラウザでログインし、トークンを保存する  
6. AI クライアントの MCP 設定に、このサーバーを登録する  

---

## 1. リポジトリの取得と移動

入手方法は配布元の案内に従ってください（ZIP の展開、または `git clone` など）。

以降、プロジェクトのルートディレクトリを `<PROJECT_ROOT>` と表記します。

```text
cd <PROJECT_ROOT>
```

---

## 2. 仮想環境とパッケージ

### Windows（PowerShell の例）

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### macOS / Linux

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

---

## 3. 環境変数ファイルの作成

プロジェクトルートに `.env` を用意します。

```text
# 例: .env.example をコピーして編集
cp .env.example .env
```

Windows のコマンドプロンプトでは `copy .env.example .env` でも構いません。

`.env` に最低限、次を設定します（値はすべて自分用のものに差し替え）。

```env
META_AUTH_TYPE=oauth
META_APP_ID=（Meta アプリのアプリ ID）
META_APP_SECRET=（Meta アプリの App Secret）
```

任意:

```env
META_AD_ACCOUNT_ID=act_（数字のみなら act_ を付けた広告アカウント ID）
META_API_VERSION=v24.0
```

**注意**

- `META_APP_SECRET` は第三者に共有しないでください。  
- `.env` は Git にコミットしないでください（通常 `.gitignore` に含まれます）。

---

## 4. Meta アプリの準備（初回のみ）

### 4.1 アプリの作成

1. Meta for Developers にログインする。  
2. 「マイアプリ」からアプリを作成する。  
3. ユースケースやタイプは README の「Meta App の作成」に沿って選ぶ（例: ビジネス向けアプリ）。  
4. アプリ名は任意（例: 社内用の識別しやすい名前）。  

### 4.2 Marketing API

アプリダッシュボードで **Marketing API** をプロダクトとして追加する。

### 4.3 アプリ ID と App Secret

「アプリ設定」→「ベーシック」から **アプリ ID** と **App Secret** を取得し、`.env` に転記する。

### 4.4 OAuth リダイレクト URI（重要）

本サーバーは OAuth 完了時に **ローカルの固定ポート** にコールバックします。

次の URI を、Meta アプリの **有効な OAuth リダイレクト URI** として登録してください。

```text
http://localhost:9876/callback
```

（コード上のポートは `9876` です。ファイアウォールや別プロセスと競合する場合は、ソースの変更が必要になるため、通常はこのまま使います。）

### 4.5 広告の作成・編集を行う場合

読み取り中心の利用でも、ツールによっては `ads_read` / `ads_management` などの権限が必要です。  
広告の作成・編集まで行う場合は、Meta の **ビジネス認証** など、プラットフォーム側の要件を満たす必要があります。詳細は Meta の公式ドキュメントを参照してください。

---

## 5. 初回認証（OAuth）

仮想環境を有効にした状態で、プロジェクトルートから次を実行します。

```bash
python meta_ads_server.py
```

- ブラウザが開き、Facebook ログインと権限の許可を求められます。  
- 成功すると、ルート付近に **トークン保存ファイル**（例: `meta_ads_token.json`）が生成されます。  
- トークンはおおむね **約 60 日** 程度で期限が切れる想定です。期限切れや検証失敗時は、同様にサーバーを起動し直して再認証できます。  
- 再認証で困る場合は、配布元の README のトラブルシューティング（トークンファイルの削除など）に従ってください。

**このコマンドは「認証用に一度動かす」用途でも使えます。** 常時起動は AI クライアント側の MCP 接続で行います。

---

## 6. AI クライアントへの接続

`<PROJECT_ROOT>` と、仮想環境内の **Python 実行ファイルのフルパス** に置き換えてください。

### 6.1 Claude Desktop

設定ファイルの場所の例:

- macOS: `~/Library/Application Support/Claude/claude_desktop_config.json`  
- Windows: `%APPDATA%\Claude\claude_desktop_config.json`  

`mcpServers` に次のようなエントリを追加します（JSON のカンマ位置に注意）。

```json
{
  "mcpServers": {
    "meta-ads": {
      "command": "<PROJECT_ROOT>/.venv/bin/python",
      "args": ["<PROJECT_ROOT>/meta_ads_server.py"]
    }
  }
}
```

Windows の例:

```json
{
  "mcpServers": {
    "meta-ads": {
      "command": "<PROJECT_ROOT>\\.venv\\Scripts\\python.exe",
      "args": ["<PROJECT_ROOT>\\meta_ads_server.py"]
    }
  }
}
```

設定後、**Claude Desktop を完全に終了してから再起動**します。

### 6.2 Claude Code（CLI）

```bash
claude mcp add meta-ads -- python <PROJECT_ROOT>/meta_ads_server.py
```

（仮想環境の `python` をフルパスで指定する方が確実な場合があります。）

### 6.3 Cursor

設定 → MCP → 新規サーバー追加:

- **名前**: 任意（例: `meta-ads`）  
- **種類**: command  
- **コマンド**: 仮想環境の `python` のフルパス  
- **引数**: `meta_ads_server.py` のフルパス  

環境変数を MCP から渡さない場合、**作業ディレクトリがプロジェクトルート**になるよう設定できるクライアントでは、そうした方が `.env` を読みやすいです。読み込まれない場合は、クライアントのドキュメントに従い `env` を指定するか、システム環境変数に設定してください。

## 7. 動作確認の例

接続後、AI に次のような依頼ができます（表現は自由です）。

- アクセス可能な広告アカウント一覧を出す  
- 直近のキャンペーン成果を要約する  
- 特定キャンペーンの配信状況を確認する  

書き込み系ツールは **既定でプレビュー（dry run）** の設計になっていることが多いです。本番反映前に AI の説明を確認してください。

---

## 8. 認証方式: システムユーザートークン（上級者向け）

長期運用やサーバー用途では、`META_AUTH_TYPE=system_user` と `META_ACCESS_TOKEN` を使う方式があります。  
トークンの発行・ローテーションは Meta Business の手順に従い、**漏洩時の影響が大きい**ため取り扱いに注意してください。

---

## 9. トラブルシューティング（よくあるもの）

| 現象 | 確認すること |
|------|----------------|
| Python が見つからない | 3.11 以上が PATH にあるか、`python` と `python3` のどちらを使うか |
| OAuth が完了しない | リダイレクト URI が `http://localhost:9876/callback` と完全一致か |
| 権限エラー | アプリの権限・ビジネス認証・広告アカウントへのアクセス権 |
| トークン無効 | 再ログイン、またはトークンファイル削除後の再認証（README 参照） |
| API バージョンエラー | `META_API_VERSION` を Meta が案内するバージョンに合わせる |

---

## 10. セキュリティの最低限のチェックリスト

- [ ] `.env` とトークン JSON をリポジトリに含めない  
- [ ] `META_APP_SECRET` をチャットやメールで送らない  
- [ ] 不要になったトークンは Meta 側で失効させる  
- [ ] 共有 PC ではトークンファイルを残さない、または暗号化ディスクを使う
